# ТЗ: харденинг инфраструктуры + AI-диалоги с аватарами (MVP)

> Статус: план следующего продуктового этапа. Он фиксирует целевую архитектуру
> и не заменяет проверку актуального production-состояния перед каждым пунктом.

Документ описывает две связанные группы работ:
- **Часть A — инфраструктура и надёжность** (то, что должно быть готово до платного трафика).
- **Часть B — AI-диалоги с аватарами** (главная новая фича MVP + новое измерение «страна»).

Продукт: Telegram Mini App + бот VocabuMe. Позиционирование — «язык для релокации» (экспаты, для MVP — Грузия). Стек: Django 5 + Postgres, `python-telegram-bot` (long polling), React (Vite), OpenAI (`gpt-5-mini`, `gpt-image-1.5`, `gpt-4o-mini-transcribe`), TTS (edge-tts/gtts).

## Зафиксированные продуктовые решения по диалогам

| Решение | Выбор |
| --- | --- |
| Режим | Полный голос: юзер говорит вслух → STT → оценка; реплики NPC озвучиваются TTS |
| Аватары | Статичные AI-портреты персонажей (генерируются один раз, хранятся локально) |
| Гейтинг | 1 бесплатный сценарий (банк), ВНЖ и work permit — Premium |
| Язык | Диалог на изучаемом (EN/KA), разбор ошибок и подсказки — на русском |
| Сценарии MVP | Открытие банковского счёта, получение ВНЖ, получение разрешения на работу |
| Страна | Грузия (`ge`), но модель данных проектируется мультистрановой |
| Источник контента | LLM «из головы» + существующие релокационные паки; ручной разбор реальных вопросов — позже |

## Порядок выполнения (фазы)

Диалоги технически **нельзя запускать поверх текущей инфраструктуры** — глобальный мьютекс на все вызовы OpenAI и dev-сервер убьют голосовой поток. Поэтому Часть A — жёсткий пререквизит.

| Фаза | Состав | Блокер для |
| --- | --- | --- |
| **P0 — Прод-надёжность** | A1 (сервер + разделение процессов), A2 (Redis + concurrency), A4 (rate-limit + cost guard) | всего платного трафика и диалогов |
| **P1 — Очереди и метрики** | A3 (Celery + вынос фоновых задач), A5 (аналитика) | масштабирования и измеримого запуска |
| **P2 — Диалоги (бэкенд)** | A6 (страна), B1–B4 (модель, контент, аватары, голосовой пайплайн, оценка) | фронта диалогов |
| **P3 — Диалоги (фронт + деньги)** | B5 (UI), B6 (гейтинг/пейволл), B7 (аналитика диалогов) | публичного релиза фичи |
| **P4 — Полировка** | B8 (паритет в боте — опционально), QA, нагрузочный тест | — |

Ориентировочная ёмкость для одного разработчика: **P0 ≈ 1 неделя, P1 ≈ 1 неделя, P2 ≈ 1.5–2 недели, P3 ≈ 1–1.5 недели, P4 — по остатку**. Итого ~4–6 недель.

# Часть A — инфраструктура и надёжность

## A1. Продакшн-сервер и разделение процессов

**Проблема.** `run.py` запускает `manage.py runserver --noreload` (dev-сервер Django) и в одном процессе держит веб + бот (polling) + reminder-loop. Падение любой части роняет всё; dev-сервер не рассчитан на прод.

**Что сделать.**
1. Ввести production WSGI-сервер: **gunicorn** с `gthread`-воркерами (много блокирующего I/O на OpenAI/TTS в request-пути).
   - Команда: `gunicorn core.wsgi:application --workers ${WEB_WORKERS:-3} --threads ${WEB_THREADS:-4} --timeout 120 --graceful-timeout 30 --bind 127.0.0.1:8000`.
2. Поставить **nginx** перед gunicorn:
   - терминирует TLS (сейчас есть `SECURE_PROXY_SSL_HEADER`);
   - отдаёт `/static/` и медиа (картинки/аудио) напрямую, минуя Django (сейчас `views.py:1085` и `:1026` отдают файлы через `FileResponse(open(...))` — это блокирует воркеры). Для приватной медиа — через `X-Accel-Redirect`.
3. Разнести процессы на **три systemd-юнита**:
   - `vocabume-web` — gunicorn;
   - `vocabume-bot` — новая management-команда `python manage.py run_bot` (выносим `run_telegram_bot` из `run.py`; polling в отдельном процессе, single-instance lock оставить);
   - `vocabume-worker` — Celery worker (см. A3); reminder-loop переезжает в Celery beat, `run.py` удаляется/депрекейтится.
4. `ALLOWED_HOSTS` (`core/settings.py:38`) вынести в env (`env_csv`), убрать хардкод IP.
5. Обновить `scripts/deploy_prod.sh`: рестартить три юнита, `collectstatic`, проверять health каждого.

**Файлы:** `core/settings.py`, `run.py` → `vocab/management/commands/run_bot.py`, новый `deploy/nginx.conf`, `deploy/*.service`, `requirements.txt` (+gunicorn), `scripts/deploy_prod.sh`.

**Критерии приёмки.**
- Прод обслуживается gunicorn за nginx; `runserver` в проде не используется.
- Рестарт/падение веба не роняет бот и наоборот.
- Статика и медиа отдаются nginx (в логах gunicorn нет запросов на `/static`, `/api/image`, `/api/audio`).
- `deploy_prod.sh` поднимает три юнита и проверяет `is-active` каждого.

## A2. Redis + управление конкурентностью OpenAI

**Проблема.** `openai_utils.openai_request_slot()` (`vocab/openai_utils.py:57`) сериализует **все** вызовы OpenAI у **всех** пользователей через один Postgres advisory-lock, с ожиданием до 180 с. Любые ~5–10 одновременных генераций/диалогов встают в один хвост → таймауты. Для голосовых диалогов это фатально.

**Что сделать.**
1. Ввести **Redis** как общую инфраструктуру (брокер очередей, rate-limit, per-user локи, кеш).
2. **Убрать глобальный advisory-lock.** Заменить на два механизма:
   - **Глобальный ограничитель RPS/конкурентности** через Redis token-bucket (`OPENAI_MAX_CONCURRENCY`, старт ~8; `OPENAI_MAX_RPS` под лимиты аккаунта). Реализовать как контекст-менеджер `openai_slot(label)` поверх Redis, а не одиночный лок.
   - **Per-user in-flight lock**: не более 1 (конфиг `OPENAI_MAX_INFLIGHT_PER_USER`) активной AI-операции на юзера — защита от спама и параллельных дорогих запросов от одного человека.
3. Все тяжёлые генерации (слова, картинки, паки) уходят в Celery (A3); ограничитель конкурентности worker'а становится естественным глобальным лимитом. В request-пути остаются только быстрые операции и шаг диалога, обёрнутые в `openai_slot`.
4. Добавить **tenacity**-ретраи с экспоненциальным бэкоффом на все вызовы OpenAI (сейчас ретраи только у `generate_image_file`).

**Файлы:** `vocab/openai_utils.py`, `core/settings.py` (Redis URL, лимиты), `requirements.txt` (+redis, +tenacity), `core/env.py`.

**Критерии приёмки.**
- Глобального одиночного лока больше нет; 10 одновременных «добавлений слова» не встают в один хвост и не упираются в 180-секундный таймаут.
- При превышении глобального лимита запросы деградируют управляемо (очередь/понятная ошибка), а не висят.
- Один пользователь не может держать более N параллельных AI-операций.

## A3. Очереди задач (Celery) и вынос фоновых задач

**Проблема.** Фоновые задачи — сырые `Thread(target=_run, daemon=True).start()` (`vocab/services.py:1172, 2280, 2606, 2686` и др.). Нет очереди, ретраев, супервизии; при рестарте in-flight задачи теряются, флаги `image_generation_in_progress` могут залипать.

**Что сделать.**
1. Подключить **Celery** (broker/result backend = Redis).
2. Перевести в задачи Celery:
   - генерацию данных слова (`generate_word_data` / batch);
   - генерацию картинок слова и черновика (`_run_word_image_generation`, `_run_draft_image_generation`);
   - подготовку паков (`manage_pack_images` / pack prepare);
   - предгенерацию TTS;
   - генерацию картинок диалога (озвучка реплик NPC — см. B4).
3. Idempotency через уже существующие `image_generation_version`.
4. **Celery beat**: перенести reminder-loop сюда; добавить периодическую задачу сброса залипших `*_in_progress` флагов (заменяет ручной `clear_stale_image_generation_flags`).
5. Заменить синхронную генерацию слова в `views.py` на enqueue + опрос статуса на фронте (у фронта уже есть polling-паттерн для картинок).

**Файлы:** новый `core/celery.py`, `vocab/tasks.py`, правки `vocab/services.py`, `vocab/views.py`, `requirements.txt` (+celery).

**Критерии приёмки.**
- Ни одного `Thread(...).start()` для генерации в request-пути.
- Рестарт воркера не теряет задачи (перезапускаются/переочередь).
- Залипшие флаги генерации автоматически сбрасываются beat-задачей.

## A4. Rate-limiting и контроль расходов на OpenAI

**Проблема.** Нет ни одного throttle на API/бот. `gpt-image-1.5` и голосовые вызовы стоят реальных денег; один скрипт-абьюзер = внезапный счёт.

**Что сделать.**
1. **Rate-limit на API** (per-user и per-IP) через Redis: разумные лимиты на дорогие эндпоинты (добавление слова, регенерация картинки, старт/шаг диалога). Ответ `429` + `Retry-After`; фронт уже умеет показывать сообщение по статусу.
2. **Дневной глобальный потолок расходов** на OpenAI (счётчик в Redis, конфиг `OPENAI_DAILY_BUDGET_USD`). При превышении — деградация: отключить генерацию картинок/диалоги, оставить базовые функции; алерт в `ALERT_CHAT_ID`.
3. **Пер-юзерные дневные лимиты AI-операций** сверх entitlements (антиабуз), в т.ч. лимиты на диалоги (см. B6).
4. **Circuit breaker** на серию ошибок OpenAI.
5. Лимиты на вход голосового диалога: длина аудио ≤ `DIALOGUE_MAX_AUDIO_SECONDS` (напр. 30 с), число ходов на сессию, число сессий/день.

**Файлы:** новый `vocab/ratelimit.py` (или `django-ratelimit`), `vocab/openai_utils.py`, `vocab/monetization.py` (лимиты), `core/settings.py`.

**Критерии приёмки.**
- Флуд запросами упирается в `429`, а не в счёт OpenAI.
- При достижении дневного бюджета включается деградация + приходит алерт.
- Аудио длиннее лимита отклоняется до вызова STT.

# Часть B — AI-диалоги с аватарами

## Обзор пользовательского потока

1. Экран «Диалоги» (новая вкладка/раздел) показывает 3 сценария с портретами персонажей и статусом (бесплатно / 🔒 Premium).
2. Юзер открывает сценарий → карточка персонажа, цель диалога («открыть счёт»), кнопка «Начать».
3. Старт: NPC произносит первую реплику (текст на изучаемом языке + озвучка TTS, автоплей).
4. Ход юзера: жмёт «микрофон», говорит вслух → аудио уходит на сервер → STT → LLM → возвращается:
   - распознанный текст юзера,
   - следующая реплика NPC (текст + аудио),
   - разбор хода на русском (что не так, как правильнее, готовая фраза «в карман»).
5. Диалог идёт до достижения цели или лимита ходов.
6. Финал: итоговый разбор на русском (сильные стороны, ошибки, фразы «в карман») + кнопка **«Добавить фразы в словарь»** (связка с SRS → retention).
7. Гейтинг: банк — 1 бесплатный проход; ВНЖ и work permit — за Premium (пейволл в нужный момент).

**Риск (зафиксировать).** Голосовой ход = STT + LLM + TTS ≈ 5–8 с задержки. Нужны явные состояния загрузки и таймауты. **Качество STT для грузинского** (`ka`) под вопросом — `gpt-4o-mini-transcribe` может плохо распознавать; для `ka` использовать `whisper-1`/`gpt-4o-transcribe` с `language="ka"` и предусмотреть **текстовый fallback-ввод**, если распознавание неуверенное.

## B2. Аватары персонажей

**Подход:** статичные AI-портреты, генерируются **один раз** офлайн (не в рантайме), кладутся в репозиторий как ассеты.

**Что сделать.**
1. Management-команда `generate_dialogue_avatars` (по аналогии с `manage_pack_images`): генерит по одному портрету на персонажа через `gpt-image-1.5`, стиль — единый (нейтральный деловой портрет, без текста, консистентная палитра как `CARD_IMAGE_STYLE_PROMPT`).
2. Сохранять как оптимизированный webp в `frontend/public/avatars/<slug>.webp` (отдаётся nginx/статикой, ноль рантайм-стоимости).
3. `avatar_slug` из конфига → URL на фронте.

**Критерии приёмки.** У каждого из 3 персонажей есть локальный аватар; в рантайме OpenAI на аватары не вызывается.

## B3. Голосовой пайплайн (STT → LLM → TTS)

**Один ход диалога** (серверная логика в `vocab/dialogue_service.py`):
1. Принять аудио (multipart) или текст (fallback).
2. **STT** в изучаемом языке (`transcribe_speech_file` сделать course-aware: `language=course_code`; для `ka` — `whisper-1`/`gpt-4o-transcribe`). Сейчас `openai_utils.py:504` хардкодит `language="en"` — исправить.
3. **LLM-ход** (`gpt-5-mini`, один вызов, JSON-ответ) — см. B4: возвращает реплику NPC + разбор + прогресс цели + флаг завершения.
4. **TTS** реплики NPC (голос из persona по курсу) → mp3, отдать URL. TTS кешируется (у `tts.py` уже есть переиспользование по хешу).
5. Обновить `DialogueSession.transcript`, `turns_count`, `goal_progress`.

Все вызовы OpenAI — через `openai_slot` (A2) и учёт бюджета (A4).

**Критерии приёмки.** Полный ход (голос → ответ с аудио) отрабатывает за приемлемое время; для `ka` есть текстовый fallback; TTS переиспользуется.

## B4. Движок оценки и промпт-дизайн

**Системный промпт хода** включает: персону и стиль NPC, цель сценария и rubric, изучаемый язык (реплики NPC — на нём), требование давать разбор **на русском**, историю диалога, инструкцию держать реплики короткими и реалистичными, мягко вести к цели и завершать при достижении цели/лимита ходов.

**Формат ответа LLM (строгий JSON):**
```json
{
  "npc_line": "реплика NPC на изучаемом языке",
  "user_eval": {
    "verdict": "ok | minor | major",
    "corrected_text": "исправленный вариант реплики юзера (или пусто)",
    "explanation_ru": "что было не так, кратко",
    "tip_ru": "как сказать лучше / чего не хватило",
    "reusable_phrase": "готовая фраза в карман (изучаемый + перевод)"
  },
  "goal_progress": {"rubric_item": true/false, ...},
  "turn_score": 0,
  "done": false
}
```

**Финальный разбор** (отдельный вызов на `finish`): `overall_score`, `strengths_ru[]`, `mistakes_ru[]`, `phrases_to_keep[]` (изучаемый + RU-перевод), `suggested_words_to_add[]` (для кнопки «Добавить в словарь»).

**Критерии приёмки.** Ответы валидны по JSON-схеме (ретрай при невалидном, как в `generate_word_data`); разбор — на русском; реплики NPC — на изучаемом языке; при достижении rubric ставится `done`.

## B7. API-контракты диалогов

Новые эндпоинты (`vocab/urls.py`, `vocab/views.py` → `vocab/dialogue_service.py`):

| Метод | Путь | Назначение |
| --- | --- | --- |
| GET | `/api/dialogue/scenarios` | список сценариев для активного курса/страны + статус блокировки |
| POST | `/api/dialogue/sessions` | старт сессии `{scenario_id}` → `{session_id, npc_line, npc_audio_url, persona, state}` (тут проверка гейтинга) |
| POST | `/api/dialogue/sessions/<id>/turn` | multipart `{audio}` или `{text}` → `{user_transcript, npc_line, npc_audio_url, feedback, turn_score, state}` |
| POST | `/api/dialogue/sessions/<id>/finish` | итог `{overall_score, strengths_ru, mistakes_ru, phrases_to_keep, suggested_words}` |
| GET | `/api/dialogue/audio/<token>` | отдача аудио реплики NPC (через nginx) |

Все — под текущей Telegram/web-авторизацией (заголовок `X-Telegram-Init-Data`, как в `api.js`), с rate-limit (A4).

**Критерии приёмки.** Контракты соответствуют фронту; ошибки/лимиты/пейволл возвращают коды, которые фронт корректно отображает.

## B8. Паритет в Telegram-боте (опционально, P4)

MVP-диалоги реализуем **только в Mini App** (голосовой UX в вебе управляемее). В боте — оставить `/subscribe` и добавить кнопку-ссылку «Открыть диалоги» в Mini App. Полноценные голосовые диалоги в чате бота (voice messages → STT) — отдельная задача после валидации.

## Ключевые риски

1. **Задержка голосового хода (5–8 с)** — обязательны состояния загрузки, таймауты, оптимистичный UI.
2. **STT для грузинского** — проверить качество на реальных фразах; иметь текстовый fallback.
3. **Стоимость** — STT+LLM+TTS на ход × попытки; дневной бюджет и лимиты (A4) обязательны до релиза.
4. **Georgian-контент** — реплики/озвучка на `ka` требуют выборочной ручной проверки перед показом.
5. **Приватность аудио** — не хранить исходные аудиофайлы юзера дольше обработки (или явный ретеншн + согласие).
